# UMO × Binar — планировщик кампании установки отопителей

## Что это
Один самодостаточный HTML-файл (вся разметка, CSS, JS и данные внутри, ноль внешних
зависимостей, работает офлайн, открывается двойным кликом). Внутренний инструмент
планирования: установка дизельных отопителей Binar на электромобили UMO 5 в сервисах
по городам РФ. Парк ~2 250–2 900 машин, окно кампании: 1 сентября — 15 ноября 2026.

Главный вопрос модели: сколько сервисов / постов / подъемников / людей нужно в каждом
городе, чтобы успеть к дедлайну, и где узкие места.

## Ключевая предметная логика (не менять без явного запроса)
- **Бамперный контур.** Лючок топливной горловины вырезается из самого заднего бампера
  (поэтому цветоподбор не нужен; допущение: цвет бампера UMO унифицирован).
  Хабы резки/сварки: Москва и Санкт-Петербург (норма хаба 3 нч на бампер, свои рабочие
  дни/смены). Регионы без хаба работают через **обменный фонд** подготовленных бамперов:
  дефолт 50 шт (настраиваемо), распределение по городам пропорционально дефициту.
  Брак 10% (настраиваемо). Цикл оборота бампера 5–13 дней.
- **Техпроцесс, 4 операции, ~12,5 нч на машину:**
  1. Снятие бампера — 60 мин × 1 механик
  2. Такт 1 (салон, проводка, Binar, экран) — 155 мин × 2 механика, без подъемника
  3. Такт 2 (бак, топливная линия, бампер, лючок) — 170 мин × 2 механика, **нужен подъемник**
  4. Такт 3 (прошивка, жидкости, прокачка) — 40 мин × 1 механик
- **Мощность сервиса** = min(по механикам, по подъемникам, по постам), с указанием
  связывающего ресурса. Выработка 85% растягивает и человеко-часы, и занятость
  подъемников/постов. Обучение: первые 10 машин × коэффициент 1,5.
  Ориентир: 1 подъемник × 8 ч × 85% ÷ 2,83 ч подъемного времени ≈ 2,4 маш/день на линию.
- **Киты (комплекты Binar) не ограничивают модель** — наоборот, модель выдает снабжению
  график потребности «город × неделя» со страховым запасом.
- **Деньги не считаем**, но структура данных должна позволить добавить ставки нормочаса позже.

## Известные результаты симуляции (для проверки, что ядро не сломано)
- Фонд = 50 → жесткое «голодание» регионов (2–5 маш/нед вместо 12).
- Фонд = 350 → голодание нулевое, к дедлайну ~92% парка.
- Брак 10% на ~900 региональных установках съедает ~90 бамперов (больше стартового фонда).
- СПб и Москва впритык: ~84 маш/нед мощности против ~87,5 требуемых.
Если после правки ядра эти ориентиры радикально «уплыли» без изменения вводных — это регрессия.

## Структура UI
Постоянная шапка: тепловая полоса кампании (сент → 15 ноя, % оснащенного парка),
KPI (оснащено / всего / темп / прогноз даты), светофор «успеваем / не успеваем».
Вкладки:
1. **Вводные** — аккордеоны параметров (парк, техпроцесс, бамперный контур, сервисы,
   календарь). Любое изменение → мгновенный пересчет всего, без кнопки «рассчитать».
   Экспорт/импорт сценария в JSON.
2. **Симуляция** — карта РФ, Play, тик = день, скраббер по неделям, города-полигоны
   наливаются градиентом, пунктирные «топливопроводы» к хабам, красный пульс на узких местах.
3. **Дашборд** — светофоры городов, недельный Gantt сервисов, «требуемый темп vs мощность»,
   загрузка хабов, динамика обменного фонда, план потребности в китах (город × неделя).
4. **План** — таблица город → сервисы (посты, подъемники, люди, график) → недельные квоты,
   плюс рекомендации по дефициту мощности.
Общее состояние — один объект сценария SCENARIO; симуляция и дашборд читают один и тот же
результат прогона, расхождений между вкладками быть не должно.

## Дизайн-система «световой тоннель»
- Фон #0A0B0D, панели #14161A, границы #26292E.
- Yandex red #FC3F1D — ТОЛЬКО действия, хабы, дефициты/критика.
- Тепловой градиент #FF2E1A → #FF8A00 → #FFC93D — ТОЛЬКО прогресс/оснащение (смысл «топливо/тепло»).
- Холодный сине-серый #7C8DA0 — состояние «до установки», тяговая батарея.
- Бейджи со skewX(-12deg), заголовки — condensed uppercase.
- Референс: `umo_binar_design_guide_v1.html`. В его шапке блок
  `<script id="design-tokens" type="application/json">` — все токены, якорные координаты
  компонентов (в пикселях пространства 1672×941) и два SVG-пути силуэта UMO 5
  (упрощенный 30 точек для мини-карточек, детальный 84 точки для hero).
  Силуэты машин — только настоящий профиль UMO 5 из этих путей, никаких «обобщенных машинок».

## Как работать (workflow)
- Разработка модульная: `engine.js` (чистое расчетное ядро, без DOM) + `ui.js` + части HTML;
  сборка в один файл конкатенацией (cat part_head.html + <script> + engine.js + ui.js ...).
  В engine.js блок `module.exports` обернут в `if (typeof module !== 'undefined')`,
  чтобы файл работал и в Node, и в браузере. Итоговый артефакт — всегда ОДИН html.
- **После каждой правки ядра:** прогнать `node test.js` (численная валидация формул
  мощности и симуляции на контрольном сценарии) и сверить с ориентирами выше.
- **После сборки:** smoke-тест через jsdom (`new JSDOM(html, { runScripts: 'dangerously' })`):
  рендер вкладок, KPI, живой пересчет на синтетических input-событиях,
  добавление/удаление сервиса. CSS-варнинги jsdom глушить `| grep -v "Could not parse CSS"`.
- Git: коммитить после каждой подтвержденной пользователем итерации, сообщения по-русски,
  осмысленные («карта: скраббер по неделям», «ядро: страховой запас китов»).
- Не переписывать файл целиком ради точечной правки. Не трогать формулы ядра при работе
  над UI и наоборот, если не просили.
- Язык общения с пользователем — русский. Пользователь дает замечания списком свободным
  текстом; проходить по всем пунктам и в конце коротко перечислять, что сделано и что
  требует его решения.
- Перед крупными переделками (новая вкладка, смена модели данных) — сначала план и вопросы,
  потом код.
