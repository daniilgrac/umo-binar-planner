# UMO × Binar — планировщик кампании установки отопителей

## Что это
Один самодостаточный HTML-файл (вся разметка, CSS, JS и данные внутри, ноль внешних
зависимостей, работает офлайн, открывается двойным кликом). Внутренний инструмент
планирования: установка дизельных отопителей Binar на электромобили UMO 5 в сервисах
по городам РФ. Парк ~2 250–2 900 машин, окно кампании: 1 сентября — 15 ноября 2026.

Главный вопрос модели: сколько сервисов / постов / подъемников / людей нужно в каждом
городе, чтобы успеть к дедлайну, и где узкие места.

## Ключевая предметная логика (не менять без явного запроса)
- **Бамперный контур = задание хабам, НЕ ограничение симуляции.** Лючок топливной
  горловины вырезается из самого заднего бампера (цветоподбор не нужен). Хабы резки/
  сварки: Москва и Санкт-Петербург; норма подготовки 2 нч на бампер, брак 10%
  (настраиваемо). Установка машин бамперами не лимитируется — наоборот, из графика
  установок модель формирует хабам поручение: недельный график «сколько бамперов
  к какой неделе готовить» (свои города — нарезка в день снятия; регионы — отгрузка
  готового за «логистику» дней до установки), требуемый **оборотный фонд** к старту
  (пик числа бамперов «в поле»), требуемые **посты** (пик подготовки × норма ÷ смена)
  и закупку новых на компенсацию брака.
- **Автоплан сервисов.** Стартовая точка: вбит только парк, реестр партнёров пуст —
  engine.planServices() сам собирает план поиска из «средних сервисов»
  (профиль настраивается в Вводных, дефолт 4 поста / 2 подъемника / 7 механиков /
  6 дн × 8 ч ≈ 3,8 маш/день, узкое — механики; подтверждено рыночным ресёрчем:
  средний СТО РФ 2–5 постов, установка Webasto 3–10 ч). Сначала закрывает требуемый
  темп, затем итеративно добавляет позиции, пока прогон не успевает к дедлайну.
  Повторный план заменяет только позиции search, работающих и переговоры не трогает.
- **Техпроцесс, 4 операции, ~12,5 нч на машину:**
  1. Снятие бампера — 60 мин × 1 механик
  2. Такт 1 (салон, проводка, Binar, экран) — 155 мин × 2 механика, без подъемника
  3. Такт 2 (бак, топливная линия, бампер, лючок) — 170 мин × 2 механика, **нужен подъемник**
  4. Такт 3 (прошивка, жидкости, прокачка) — 40 мин × 1 механик
- **Мощность сервиса** = min(по механикам, по подъемникам, по постам), с указанием
  связывающего ресурса. Выработка 85% растягивает и человеко-часы, и занятость
  подъемников/постов. Обучение: первые 10 машин × коэффициент 1,5.
  Ориентир: 1 подъемник × 8 ч × 85% ÷ 2,83 ч подъемного времени ≈ 2,4 маш/день на линию.
- **Киты (комплекты Binar) не моделируются вовсе** — поставки подстраиваются под график
  машин. Графиком потребности для снабжения служат недельные квоты на вкладке «План»
  (кит должен лежать на складе сервиса к началу недели установки).
- **Деньги не считаем**, но структура данных должна позволить добавить ставки нормочаса позже.
- **Сознательные упрощения** (не возвращать без запроса): старт кампании зафиксирован
  01.09.2026 — под него зашит график поставок машин; горизонт симуляции = дедлайн + 6 недель;
  двор хабовых городов (10 мест) и праздник 4 ноября — константы ядра; смен у сервисов нет —
  только «часов в день» (вторая смена = 16 ч); курьерская логистика внутри города с хабом
  не моделируется.

## Известные результаты симуляции (для проверки, что ядро не сломано)
Контрольный сценарий тестов — «исторический сид» из 24 работающих линий
(2 поста/1 подъемник/5 механиков; Мск 3, СПб 7, НН 2, Екб 2, остальные по 1 — хелпер в test.js):
- К дедлайну ~92,3% парка, финиш 25.11 (после дедлайна).
- СПб впритык: 84 маш/нед против 87,5 требуемых; Москва тоже (36 против 36,8).
- Задание хабам: пик ~47 бамп/день → 12 постов, оборотный фонд ~148 шт,
  брак 10% на ~900 региональных ≈ 90 шт закупки.
Автоплан от пустого реестра (дефолтный парк): 20 позиций поиска (СПб 5, Мск 2, НН 2,
остальные по 1), 100% парка к дедлайну, финиш ~05.11; задание: пик ~62 бамп/день →
16 постов, фонд ~195. Средний сервис: 3,81 маш/день (узкое — механики), 22,8 маш/нед.
Если после правки ядра эти ориентиры радикально «уплыли» без изменения вводных — это регрессия.

## Назначение и workflow пользователя
Инструмент технико-операционной команды (главный инженер). Рабочий цикл:
вбить парк и дедлайн → модель САМА строит план поиска из «средних сервисов»
(сколько партнёров и каких в каждом городе) и задание хабам по бамперам →
по мере переговоров статусы меняются (надо найти → в переговорах → работает) →
два светофора в шапке показывают цену незакрытого поиска → план выгружается
в .xlsx как поручение: поиск партнёров + задание хабам + квоты.

## Структура UI
Постоянная шапка: тепловая полоса кампании, KPI (оснащено / всего / темп / прогноз /
хабы), **два светофора сценариев** — «Факт» (только работающие партнёры) и «С поиском»
(+ переговоры + позиции поиска) — и переключатель, какой сценарий показывают все вкладки.
У партнёра есть статус `active | talks | search`; runSim принимает
`{include: Set(статусов)}`, признанные ориентиры — по полному прогону (все статусы).
Вкладки:
1. **Вводные** — нумерованные аккордеоны: 1 дедлайн → 2 парк → 3 техпроцесс →
   4 бамперный контур и хабы (только норма/брак/график хаба — фонд и посты теперь
   расчётные) → 5 средний сервис-партнёр (профиль для автоплана); плюс свернутые
   «Тонкие настройки» и «Проверка расчёта мощности». У каждой секции — строка «зачем
   она модели». Любое изменение → мгновенный пересчет. Экспорт/импорт сценария в JSON.
2. **Партнёры и план** — реестр партнёров по городам со статусами и колор-кодингом
   (работает = тёплый, переговоры = холодный, найти = красный пунктир); кнопка
   «Пересчитать план поиска» (planServices) и «+ искомые» у города с дефицитом;
   колонка «старт не позже» — engine.latestStart, пробные прогоны по понедельникам;
   рекомендации; секция «Задание хабам» (сводка фонд/посты/брак + недельная таблица
   подготовки по городам из taskCW); недельные квоты (они же график потребности
   в китах); выгрузка .xlsx — 5 листов (Поиск партнёров / Реестр / Бампера — задание
   хабам / Квоты / Вводные), zip и OOXML пишутся руками в ui.js (zipStore + sheetXML).
3. **Дашборд** — светофоры городов, недельный Gantt сервисов, «требуемый темп vs мощность»,
   загрузка хабов, динамика обменного фонда.
4. **Симуляция** — лента критических событий (engine.analyzeEvents: голодания городов,
   перегруз хабов, укомплектованные города, дедлайн; клик — прыжок карты к дню) +
   стилизованная SVG-схема РФ (координаты SIM_XY в ui.js, гранёные пятиугольники,
   тепловой налив, топливопроводы, красный пульс узких мест), Play (тик = день), скраббер.
Общее состояние — один объект SCENARIO; прогоны SIM_FACT / SIM_PLAN, выбранный — SIM.
Дефициты и рекомендации всегда считаются по SIM_PLAN, светофоры — по обоим.

## Дизайн-система «световой тоннель»
- Фон #0A0B0D, панели #14161A, границы #26292E.
- **Контраст смягчён осознанно** (просьба пользователя): основной текст #A6ACB4 (~8:1),
  вторичный #8A9099 (~5,5:1), тёплый акцент текста #D9AC3B; насыщенные #FFC93D/#FF8A00 —
  только в градиентных заливках, не в тексте. Залитые красные плашки — #CE3212 (--actionBg),
  чтобы белый текст на них проходил AA; #FC3F1D остаётся для текста/линий на тёмном.
  Не возвращать белый #F4F5F7 в текст. После правок палитры прогонять `node contrast.js` —
  аудит WCAG AA (все пары ≥ 4,5:1, основной текст ≤ 10:1).
- Yandex red #FC3F1D — ТОЛЬКО действия, хабы, дефициты/критика.
- Тепловой градиент #FF2E1A → #FF8A00 → #FFC93D — ТОЛЬКО прогресс/оснащение (смысл «топливо/тепло»).
- Холодный сине-серый #7C8DA0 — состояние «до установки», тяговая батарея.
- **Колор-кодинг ввод/вывод:** холодные сине-серые поля и подписи — редактируемые вводные
  («сырьё» вносит человек), тёплые значения — расчёт модели. Легенда — вверху «Вводных».
- Бейджи со skewX(-12deg), заголовки — condensed uppercase.
- Референс: `umo_binar_design_guide_v1.html`. В его шапке блок
  `<script id="design-tokens" type="application/json">` — все токены, якорные координаты
  компонентов (в пикселях пространства 1672×941) и два SVG-пути силуэта UMO 5
  (упрощенный 30 точек для мини-карточек, детальный 84 точки для hero).
  Силуэты машин — только настоящий профиль UMO 5 из этих путей, никаких «обобщенных машинок».

## Как работать (workflow)
- Разработка модульная: `engine.js` (чистое расчетное ядро, без DOM) + `ui.js` + части HTML;
  сборка в один файл конкатенацией (cat part_head.html + <script> + engine.js + ui.js ...).
  В engine.js блок `module.exports` обернут в `if (typeof module !== 'undefined')`,
  чтобы файл работал и в Node, и в браузере. Итоговый артефакт — всегда ОДИН html.
- **После каждой правки ядра:** прогнать `node test.js` (численная валидация формул
  мощности и симуляции на контрольном сценарии) и сверить с ориентирами выше.
- **После сборки:** smoke-тест через jsdom (`new JSDOM(html, { runScripts: 'dangerously' })`):
  рендер вкладок, KPI, живой пересчет на синтетических input-событиях,
  добавление/удаление сервиса. CSS-варнинги jsdom глушить `| grep -v "Could not parse CSS"`.
- Публикация: GitHub Pages — https://daniilgrac.github.io/umo-binar-planner/ (репо
  daniilgrac/umo-binar-planner, ветка main, index.html = копия артефакта из build.sh).
  `git push` в main обновляет страницу через минуту-две. Репозиторий публичный.
- Git: коммитить после каждой подтвержденной пользователем итерации, сообщения по-русски,
  осмысленные («карта: скраббер по неделям», «ядро: страховой запас китов»).
- Не переписывать файл целиком ради точечной правки. Не трогать формулы ядра при работе
  над UI и наоборот, если не просили.
- Язык общения с пользователем — русский. Пользователь дает замечания списком свободным
  текстом; проходить по всем пунктам и в конце коротко перечислять, что сделано и что
  требует его решения.
- Перед крупными переделками (новая вкладка, смена модели данных) — сначала план и вопросы,
  потом код.
